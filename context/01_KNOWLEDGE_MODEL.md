# 知识模型

## 一、知识点清单

从五个学习目标倒推，穷举读者需要掌握的知识点。每个知识点标注 ID 以便在依赖图中引用。

### 域 A：基础约束

读者必须先建立"分布式系统为什么难"的直觉，才能理解后续所有方案的存在理由。

| ID | 知识点 | 说明 |
|----|--------|------|
| A1 | 分布式系统的定义 | 多台通过网络通信的计算机协作完成任务；与单机程序的本质区别 |
| A2 | 为什么需要分布式 | 可用性、扩展性、地理分布——不是选择题，是被迫的 |
| A3 | 网络不可靠性 | 消息可能丢失、延迟、重复、乱序；网络分区是常态而非异常 |
| A4 | 无全局时钟 | 不同节点的物理时钟不同步，无法用时间戳判定事件先后 |
| A5 | 部分故障 | 系统的一部分故障而其余正常运行；与单机"要么全好要么全坏"的根本差异 |
| A6 | 故障模型 | 崩溃停止（crash-stop）、崩溃恢复（crash-recovery）；本教材不涉及拜占庭 |
| A7 | 超时与不确定性 | 收不到响应时，无法区分"对方崩溃"和"网络延迟"——这是大量设计困境的根源 |
| A8 | 两将军问题 | 在不可靠信道上无法达成确定性一致——作为"共识很难"的早期直觉铺垫 |

### 域 B：时间与顺序

分布式系统没有全局时钟，如何给事件排序是核心问题。这个域为一致性模型和共识提供数学工具。

| ID | 知识点 | 说明 |
|----|--------|------|
| B1 | 物理时钟及其局限 | 时钟漂移、NTP 的精度上限；为什么不能用物理时间戳判定因果 |
| B2 | 事件的因果关系 | 如果事件 a 可能影响了事件 b，则 a 因果先于 b |
| B3 | happened-before 关系 | Lamport 的形式化定义：进程内顺序 + 消息发送→接收 + 传递性 |
| B4 | 偏序与全序 | 并发事件之间没有因果关系，因此事件集只有偏序；全序需要额外机制 |
| B5 | Lamport 时钟 | 给事件分配递增编号，保留因果序（a→b 则 L(a)<L(b)），但反之不成立 |
| B6 | 向量时钟 | 精确捕获因果关系：V(a)<V(b) 当且仅当 a→b；用于检测并发写入 |
| B7 | 全序广播 | 所有节点以相同顺序接收所有消息；与共识等价的关键桥梁 |

### 域 C：复制

复制是分布式数据系统最基础的能力。复制带来的问题（延迟、冲突、不一致）是后续一致性讨论的直接动机。

| ID | 知识点 | 说明 |
|----|--------|------|
| C1 | 复制的动机 | 可用性（节点挂了还能服务）、降低延迟（就近读取）、读扩展 |
| C2 | 单主复制 | 一个节点接受写入，异步或同步复制到从节点 |
| C3 | 同步复制 vs 异步复制 | 同步：强一致但慢且脆弱；异步：快但可能丢数据。核心权衡 |
| C4 | 复制延迟及其症状 | 异步复制必然有延迟——"刚写入就读不到"的根源 |
| C5 | 读写一致性 | read-your-writes：用户应能读到自己刚写入的数据 |
| C6 | 单调读 | 用户不应看到时间倒流（先看到新值再看到旧值） |
| C7 | 一致前缀读 | 如果事件 A 导致了 B，任何观察者看到 B 时都应已看到 A |
| C8 | 主从切换与故障转移 | 主节点崩溃后如何选新主？手动、预定义、自动选举——引出"选举需要共识"的动机 |
| C9 | 多主复制 | 多个节点都接受写入——解决单主的可用性瓶颈，但引入写冲突 |
| C10 | 写冲突 | 两个节点同时修改同一数据：如何检测、如何解决 |
| C11 | 冲突解决策略 | LWW（last-writer-wins）、合并、应用层自定义解决 |
| C12 | 无主复制与 quorum | 无固定主节点，W+R>N 保证读到最新值；灵活但一致性保证有限 |
| C13 | 反熵与读修复 | 后台同步 + 读时发现并修复不一致——quorum 系统的辅助机制 |

### 域 D：一致性模型

一致性模型定义了"系统对外表现的行为约束"。这是本教材的核心难点之一，需要密集的推导节点。

| ID | 知识点 | 说明 |
|----|--------|------|
| D1 | 什么是一致性模型 | 对"多副本系统对外行为"的约束规范；区分 ACID 的 C、CAP 的 C |
| D2 | 最终一致性 | 如果停止写入，最终所有副本收敛到相同状态 |
| D3 | 因果一致性 | 有因果关系的操作，所有节点看到的顺序相同；并发操作顺序可以不同 |
| D4 | 线性一致性 | 系统表现得像只有一个副本，所有操作都有全局顺序且符合实时约束 |
| D5 | 顺序一致性 | 所有操作有全局顺序，但不要求符合实时约束（比线性一致性弱） |
| D6 | 一致性模型的谱系 | 从弱到强：最终一致性 → 因果一致性 → 顺序一致性 → 线性一致性 |
| D7 | 强一致性的代价 | 越强的一致性 = 越高的延迟 + 越低的可用性；这是物理定律而非实现缺陷 |
| D8 | CAP 定理 | 网络分区发生时，一致性和可用性不可兼得；以及这个定理经常被误解的地方 |

### 域 E：共识

共识是分布式系统最核心的问题。从"为什么需要共识"到"Raft 如何工作"需要最密集的推导链。

| ID | 知识点 | 说明 |
|----|--------|------|
| E1 | 共识问题的定义 | 多个节点对某个提议值达成一致：终止性、一致性、合法性 |
| E2 | FLP 不可能性 | 异步系统中即使只有一个节点可能崩溃，也无法保证共识一定终止 |
| E3 | 绕过 FLP 的实践策略 | 引入超时/随机化/部分同步假设，在实践中可以工作 |
| E4 | Raft：领导者选举 | 任期、候选人、投票规则——如何在节点中选出一个领导者 |
| E5 | Raft：日志复制 | 领导者将操作追加到日志并复制到多数节点，确认后提交 |
| E6 | Raft：安全性 | 选举限制（日志至少一样新才能当选）保证已提交的条目不会丢失 |
| E7 | 共识与全序广播的等价性 | 共识可实现全序广播，全序广播可实现共识——互为基础设施 |
| E8 | 共识与线性一致性的关系 | 线性一致性的 CAS 操作需要共识；共识是实现强一致性的必要工具 |
| E9 | 共识在真实系统中的应用 | ZooKeeper（ZAB）、etcd（Raft）：配置管理、领导者选举、分布式锁 |

### 域 F：容错

容错贯穿整个教材，但这里集中处理故障检测和消息可靠性——复制和共识都依赖这些基础设施。

| ID | 知识点 | 说明 |
|----|--------|------|
| F1 | 故障检测：心跳与超时 | 定期发送心跳，超时未收到则怀疑故障 |
| F2 | 不完美故障检测器 | 不可能同时满足完全性（最终检测到所有故障）和准确性（不误判）|
| F3 | 消息语义 | at-most-once、at-least-once、exactly-once 的定义和区别 |
| F4 | 幂等性 | 重复执行产生相同结果；使 at-least-once 在效果上等同于 exactly-once |
| F5 | 重试策略 | 立即重试、退避重试、指数退避；与幂等性配合使用 |

### 域 G：分区

当数据量或吞吐超过单机上限，需要将数据分片到多台机器。分区引入跨分区操作的困难。

| ID | 知识点 | 说明 |
|----|--------|------|
| G1 | 分区的动机 | 单机容量或吞吐不够时，水平拆分数据 |
| G2 | 分区策略 | 哈希分区（均匀分布）vs 范围分区（支持区间查询）|
| G3 | 热点问题 | 数据或请求分布不均导致某些分区过载 |
| G4 | 再平衡 | 增减节点时如何重新分配分区，最小化数据搬迁 |
| G5 | 分区中的次级索引 | 本地索引 vs 全局索引：查询效率与维护成本的权衡 |
| G6 | 请求路由 | 客户端如何知道数据在哪个分区：客户端路由、代理路由、协调服务 |

### 域 H：分布式事务

跨越多节点或多分区的操作如何保证原子性。

| ID | 知识点 | 说明 |
|----|--------|------|
| H1 | 单机事务回顾 | ACID 四个属性在单机上的含义，作为对比基线 |
| H2 | 分布式事务的需求 | 跨分区写入、跨服务操作——为什么单机事务的方法不够 |
| H3 | 两阶段提交（2PC） | 准备阶段 + 提交阶段：协调者确保所有参与者要么全提交要么全回滚 |
| H4 | 2PC 的阻塞问题 | 协调者崩溃后参与者进退两难——2PC 不是容错的 |
| H5 | 分布式锁与租约 | 用共识服务（ZooKeeper/etcd）实现锁；租约的自动过期机制 |
| H6 | Saga 模式 | 拆分为本地事务序列 + 补偿操作；最终一致性替代强一致性 |

## 二、知识域划分

八个域按内聚程度划分如下，域内知识点有较强依赖链，域间依赖相对稀疏。

```
A 基础约束 ─────────────────────────────────┐
│                                            │
├── F 容错（故障检测、消息语义）               │  基础层
│                                            │  （被所有后续域依赖）
├── B 时间与顺序（逻辑时钟、因果序）           │
│                                            │
├───────────────────────────────────────────┘
│
├── C 复制 ──── D 一致性模型 ──── E 共识       ← 核心推导链
│                                               （三个域串联，是教材主干）
├── G 分区
│
└── H 分布式事务
```

**核心推导链（C → D → E）**是整个教材的主干：复制引出一致性问题，一致性问题引出共识需求。这条链上的推导节点密度最高，也是教学难度最大的部分。

## 四、实践锚点

在依赖图中标注"学完这组理论后足以支撑一个有意义的实践任务"的节点。遵循"宁缺毋滥"原则——每个实践都连接真实世界，完成后读者能对照真实系统理解其内部机制。

### 实践锚点 1：观察分布式系统的三个基础约束

**前置知识**：A1–A8, F1, F3

**实践内容**：在一个模拟环境中运行简单的客户端-服务端系统，依次注入网络延迟、消息丢失、节点崩溃，观察系统行为如何偏离"单机直觉"。读者亲眼看到：超时不代表失败、消息可能重复到达、部分节点故障不影响其他节点。

**真实世界连接**：对照微服务调用超时、重试风暴等日常场景。

### 实践锚点 2：实现单主复制，体验复制延迟

**前置知识**：C1–C7, F1

**实践内容**：实现一个单主复制的 KV 存储（一主两从，异步复制）。先验证正常写入和读取，然后引入网络延迟，观察"刚写入就读不到"的现象。接着实现 read-your-writes 保证，验证修复效果。

**真实世界连接**：对照 MySQL/PostgreSQL 主从复制延迟、Redis 主从延迟。读者理解为什么"加个 sleep"不是正确的解决方案。

### 实践锚点 3：用向量时钟检测并发写入冲突

**前置知识**：C9–C11, B6

**实践内容**：在多主复制场景下，用向量时钟检测并发写入。读者实现冲突检测逻辑，观察 LWW 会静默丢弃数据，然后实现一种保留冲突版本的策略。

**真实世界连接**：对照 DynamoDB 的冲突解决、CouchDB 的版本树。

### 实践锚点 4：实现 Raft 共识协议

**前置知识**：E1–E6（以及所有前置）

**实践内容**：渐进式实现 Raft 的三个子问题——领导者选举 → 日志复制 → 安全性保证。每步有独立的正确性测试。这是全教材最核心的实践，前面所有理论在此汇聚。

**真实世界连接**：对照 etcd（Raft）和 ZooKeeper（ZAB）。完成后读者能读懂这些系统的一致性保证文档，理解为什么"Raft 领导者挂了之后有短暂不可用"。

### 实践锚点 5：构建分区容错的 KV 存储

**前置知识**：G1–G6, E9, H2–H4

**实践内容**：在实践锚点 4 的 Raft KV 基础上，加入哈希分区。实现分区分配、请求路由、跨分区读写。引入节点增减，观察再平衡过程。

**真实世界连接**：对照 Redis Cluster 的哈希槽分配、Kafka 的分区再平衡。读者理解为什么"增加节点不是无代价的"。

## 六、设计说明

### 1. "日常开发映射"不设独立知识域

Overview 要求覆盖"上述概念在日常开发中的映射（缓存、消息队列、微服务通信、分布式锁）"。本模型不将其设为独立的知识域，而是通过两个机制实现覆盖：

- **症状驱动入口**：每个理论概念从日常场景切入（如 C4 从"刚写入就读不到"出发，F3 从"消息为什么会重复"出发），映射关系融入教学过程本身。
- **实践锚点的真实世界连接**：每个实践都明确对照 Redis、Kafka、ZooKeeper 等真实系统。

这样做的原因：日常映射是教学方法而非独立知识——把它从推导链中抽出来单独讲，反而会打断"症状→机制→方案"的推导节奏。

### 2. 跨域依赖的课程编排约束

C11（冲突解决）依赖 B6（向量时钟）。这意味着课程编排时，域 B 的逻辑时钟部分（B1–B6）必须安排在域 C 的多主复制部分（C9–C11）之前。一种可行方案：在单主复制（C2–C7）之后、多主复制之前插入时间与顺序的课程。

### 3. 一致性模型的教学顺序

D4（线性一致性）安排在 D5（顺序一致性）之前，先强后弱。理由：线性一致性匹配读者"系统像只有一个副本"的天然直觉，容易建立初始理解。顺序一致性则通过"放松线性一致性的实时约束"来定义，作为对比更容易理解。如果先教弱模型，读者缺少对照物，反而更困惑。

### 4. C8（主从切换）作为 C→E 的桥梁

新增 C8 填补了从复制到共识的动机缺口。单主复制中主节点崩溃后"谁来当新主"的问题，是读者第一次自然感受到"需要节点之间达成一致"——这正是共识问题的直觉入口。C8 不深入共识算法（那是域 E 的职责），只展示问题的存在和手动/简单方案的局限性。
